%%% Proof source.
%%%
%%% The game is written as a flat sequence of \State lines (no \Procedure
%%% wrapper), so every \tfsegment marker sits at block depth 0 --- the
%%% placement cropping requires. Four \tfsegment markers cut the body into
%%% these pieces:
%%%   * segment 0: the \begin{algorithmic} opener + game header  (always kept)
%%%   * Setup, Encapsulation, Key derivation, DEM+output  (interior --- each
%%%     collapses to a stub in any diffed render where it is unchanged)
%%%   * the bare \end{algorithmic} closer  (final segment --- always kept)
%%% Only the opener and the closer are pinned; every content segment can crop.
\begin{tfsource}{kemdem}
\begin{algorithmic}[1]
\State \textbf{Game} $\tfgamename{G0}$: \quad $b \getsr \{0,1\}$;\enspace receive $(m_0, m_1)$ from $\Adversary$
\tfsegment{Setup}
\State $(\pk, \sk) \gets \KeyGen()$
\tfsegment{Encapsulation}
\tfonly{G0,G2-G4}{\State $(c_1, K) \gets \Encaps(\pk)$}
\tfonly{G1}{\State $(c_1, \hat{K}) \gets \Encaps(\pk)$}
\tfonly{G1}{\State $K \getsr \mathcal{K}$}
\tfsegment{Key derivation}
\State $r \getsr \{0,1\}^\lambda$
\tfonly{G0,G1}{\State $\mathit{pad} \gets \mathrm{PRF}(K, r)$}
\tfonly{G2}{\State $\mathit{pad} \gets \RF(r)$}
\tfonly{G3,G4}{\State $\mathit{pad} \getsr \{0,1\}^{\ell}$}
\tfsegment{DEM}
\tfonly{G0-G3}{\State $c_2 \gets \mathit{pad} \oplus m_b$}
\tfonly{G4}{\State $c_2 \getsr \{0,1\}^{\ell}$}
\State \Return $(c_1, c_2)$ to $\Adversary$
\end{algorithmic}
\end{tfsource}

\section{What this example demonstrates}

Each game hop below changes exactly one line, in one segment, of an otherwise
unchanging listing. With \verb|\tfcropdefault{on}| set in the preamble, every
diffed \verb|\tfrendergame| crops away the segments that did not change,
collapsing each of them into a single \verb|\tfsegmentstub| line such as
\textcolor{gray}{[Setup unchanged]}. Only the parts that anchor a balanced,
compilable listing are pinned: segment~0 (the \verb|algorithmic| opener and the
game header) and the final segment (the bare \verb|\end{algorithmic}| closer).
Every content segment in between---\emph{Setup}, \emph{Encapsulation},
\emph{Key derivation}, and \emph{DEM}---collapses whenever the hop leaves it
unchanged.

Compare Game~$\tfgamename{kemdem}{G4}$ below (three collapsed stubs) against the
same render forced to a full listing with the PDF-only \verb|crop=off| key.

\section{Construction}

Let $(\KeyGen, \Encaps, \mathsf{Decaps})$ be a KEM with key space $\mathcal{K}$
and let $\mathrm{PRF}\colon \mathcal{K} \times \{0,1\}^\lambda \to \{0,1\}^{\ell}$
be a pseudorandom function. The hybrid encryption of a message $m$ encapsulates
a key $K$, derives a one-time pad $\mathit{pad} \gets \mathrm{PRF}(K, r)$ for a
fresh nonce $r$, and sends $(c_1, r, \mathit{pad} \oplus m)$.

\section{Security}

\begin{theorem}
If the KEM is $\INDCPA$-secure and $\mathrm{PRF}$ is a secure pseudorandom
function, then the hybrid scheme is $\INDCPA$-secure.
\end{theorem}

listing of every game side by side; the individual figures that follow are
